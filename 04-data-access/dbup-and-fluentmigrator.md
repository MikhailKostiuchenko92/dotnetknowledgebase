# DbUp and FluentMigrator

**Category:** Data Access / Migrations & Schema Management
**Difficulty:** 🟡 Middle
**Tags:** `DbUp`, `FluentMigrator`, `SQL-migrations`, `schema-management`, `CI/CD`, `EF Core`

## Question

> What are DbUp and FluentMigrator? How do they differ from EF Core migrations? When would you choose a SQL-script–based migration tool over EF Core's code-first approach?

## Short Answer

**DbUp** executes plain SQL scripts in versioned order — scripts are embedded as resources, tracked in a journal table, and run once. It is migration-script–based: you write T-SQL, and DbUp runs it. **FluentMigrator** is a code-first migration framework (like EF Core Migrations but without an ORM) — migrations are C# classes with `Up()`/`Down()` methods and a fluent DDL DSL. Both are alternatives to EF Core migrations when you want full control over the SQL, don't use EF Core as your ORM, or have DBAs who own the schema. EF Core migrations win when your model and schema change together in lock-step and you want automatic migration generation from model diffs.

## Detailed Explanation

### DbUp — SQL Script–Based Migrations

DbUp (NuGet: `dbup-sqlserver`) treats each `.sql` file as a migration unit. Scripts are run once and tracked by filename in a journal table.

```csharp
// Program.cs — apply migrations at startup or as a pre-deployment step
var upgrader = DeployChanges.To
    .SqlDatabase(connectionString)
    .WithScriptsEmbeddedInAssembly(Assembly.GetExecutingAssembly(),
        s => s.StartsWith("Migrations."))  // filter by namespace/prefix
    .WithTransactionPerScript()  // roll back a failed script atomically
    .LogToConsole()
    .Build();

var result = upgrader.PerformUpgrade();

if (!result.Successful)
{
    Console.Error.WriteLine(result.Error.Message);
    Environment.Exit(1);
}
```

```sql
-- Migrations/0001_CreateOrders.sql (embedded resource)
CREATE TABLE Orders (
    Id       INT IDENTITY PRIMARY KEY,
    CustomerId INT NOT NULL,
    Total    DECIMAL(18,2) NOT NULL,
    Status   NVARCHAR(20) NOT NULL DEFAULT 'Pending',
    CreatedAt DATETIME2 NOT NULL DEFAULT GETUTCDATE()
);
```

Scripts are **named with a versioning prefix** (e.g., `0001_`, timestamp, or date-based) to define execution order.

### FluentMigrator — C# DSL for Schema Changes

FluentMigrator (NuGet: `FluentMigrator`, `FluentMigrator.Runner.SqlServer`) provides a code-first DSL similar to EF Core but without ORM coupling:

```csharp
[Migration(20240315_001)]
public class AddStatusColumn : Migration
{
    public override void Up()
    {
        Alter.Table("Orders")
            .AddColumn("Reference")
            .AsString(50)
            .Nullable()
            .WithDefaultValue("");
    }

    public override void Down()
    {
        Delete.Column("Reference").FromTable("Orders");
    }
}
```

```csharp
// Program.cs — register FluentMigrator runner
services.AddFluentMigratorCore()
    .ConfigureRunner(rb => rb
        .AddSqlServer()
        .WithGlobalConnectionString(connStr)
        .ScanIn(Assembly.GetExecutingAssembly()).For.Migrations())
    .AddLogging(lb => lb.AddFluentMigratorConsole());

// Apply migrations
var runner = services.BuildServiceProvider().GetRequiredService<IMigrationRunner>();
runner.MigrateUp();
```

### Comparison Table

| Feature | EF Core Migrations | DbUp | FluentMigrator |
|---------|-------------------|------|----------------|
| Migration generation | Automatic (model diff) | Manual (SQL scripts) | Manual (C# DSL) |
| Language | C# | SQL | C# |
| ORM coupling | Yes (requires EF Core) | None | None |
| Rollback (`Down`) | Auto-generated | Manual (separate script) | Manual `Down()` |
| DBA-friendly | Low (need C# knowledge) | ✅ (pure SQL) | Medium |
| Multi-DB support | ✅ (via providers) | ✅ | ✅ |
| Complex T-SQL (temp tables, cursors) | Awkward (raw SQL required) | ✅ Native | Via Execute.Sql() |
| Journey table | `__EFMigrationsHistory` | `SchemaVersions` | `VersionInfo` |

### When to Choose DbUp / FluentMigrator over EF Core Migrations

- **DBA-owned schemas**: DBAs write and review SQL directly — EF Core's C# migration format is not DBA-friendly
- **Existing database**: you have a large existing schema not generated by EF Core — reverse-engineering a snapshot is fragile
- **Complex migrations**: stored procedures, views, functions, triggers — EF Core migrations handle these awkwardly
- **No EF Core**: using Dapper, ADO.NET, or no ORM — EF Core migrations require EF Core packages
- **Script audit trail**: SQL scripts in source control are more transparent for compliance audits than EF Core's C# code
- **Multi-database support**: same scripts target SQL Server, PostgreSQL, and MySQL — DbUp and FluentMigrator support all via providers

## Code Example

```csharp
// DbUp with environment-specific scripts and pre/post-deployment
var upgrader = DeployChanges.To
    .SqlDatabase(connectionString)
    // Run scripts in order: 0001, 0002, etc.
    .WithScriptsEmbeddedInAssembly(
        typeof(Program).Assembly,
        script => script.Contains(".Schema."))
    // Seed data scripts run separately, are always run (not journaled)
    .WithScriptsEmbeddedInAssembly(
        typeof(Program).Assembly,
        script => script.Contains(".SeedData."),
        new SqlScriptOptions { ScriptType = ScriptType.RunAlways })
    .WithTransactionPerScript()
    .JournalToSqlTable("dbo", "MigrationHistory")  // custom journal table name
    .LogToConsole()
    .Build();

// Check for pending scripts without applying (CI validation)
if (!upgrader.IsUpgradeRequired())
{
    Console.WriteLine("No pending migrations");
    return 0;
}
```

## Common Mistakes / Pitfalls

- **Renaming a DbUp script file**: DbUp identifies scripts by their filename — renaming a script that was already applied breaks the journal (DbUp thinks it's a new script). Never rename applied scripts.
- **Not embedding scripts as resources**: DbUp reads scripts from the assembly by default. Forgetting to set `Build Action = Embedded Resource` causes scripts to not be found.
- **Skipping `Down()` in FluentMigrator**: FluentMigrator allows empty `Down()`, but rollback becomes impossible. Always implement `Down()` for column/table additions; for destructive changes, document why rollback is not supported.
- **Using DbUp for application startup in production**: just like EF Core, running DbUp in `Program.cs` on a 100M-row table risks startup timeouts. Run DbUp as a separate pre-deployment step.
