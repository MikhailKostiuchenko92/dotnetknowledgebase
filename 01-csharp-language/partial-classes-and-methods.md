# Partial Classes and Methods

**Category:** C# / OOP in C#
**Difficulty:** Middle
**Tags:** `partial`, `source-generators`, `generated-code`, `partial-methods`, `codegen`

## Question

> What are partial classes and partial methods in C#, and why are they useful?

Also asked as:
- "Why do generated files often use `partial` classes?"
- "What is a partial method, and how has it changed in modern C#?"
- "How do source generators and user code cooperate through partial types?"

## Short Answer

`partial` lets one type or member declaration be split across multiple files, with the compiler combining the pieces into a single final type. This is especially useful for generated code plus hand-written code, because each side can evolve separately without merge conflicts. Partial methods are lightweight hooks inside partial types; modern C# allows richer partial methods than the old "private void only" model, which made them far more useful for source generators.

## Detailed Explanation

### Why Partial Types Exist

A partial class, struct, interface, or record lets you divide a single logical type into multiple source files. The compiler merges all parts as long as they share the same namespace, type name, generic arity, and assembly.

The classic use case is generated code. One file is generated by a tool, while another file contains user customization. That avoids editing generated files directly.

| Pattern | Benefit |
|---|---|
| Generated part + user part | Safe regeneration without losing custom logic |
| Large UI or designer type | Better file organization |
| Source generator output | Compile-time augmentation without reflection |

### Partial Classes and Other Partial Types

Most developers first meet `partial class`, but `partial` also works with structs, interfaces, and records. All pieces are merged into one resulting type, so fields, methods, attributes, base interfaces, and nested types combine according to the language rules.

This does not create two runtime types. It creates one runtime type built from many source fragments.

### Partial Methods

Partial methods are declarations inside partial types that another part may implement. Historically they were very restricted:

- implicit `private`
- `void` only
- no `out` parameters
- removable by the compiler when unimplemented

Modern C# relaxed that model. Partial methods can now have accessibility and non-void return types, but if they are not erasable-style private hooks, they must be implemented.

That made them much more useful for source generators. A generator can emit one partial declaration, and user code can implement the companion member.

> **Tip:** Think of partial methods as compile-time extension points inside a type you control, not as a general replacement for interfaces or events.

### Generated Code and Source Generators

Source generators commonly emit partial members to avoid runtime reflection and preserve strong typing. Examples in modern .NET include generated regexes, JSON serialization contexts, and logging helpers.

A typical pattern is:

1. developer writes a partial type or partial method signature
2. generator emits the missing implementation in another file
3. compiler merges them

That gives AOT-friendly, compile-time generated code while keeping the authoring experience simple.

### When Not to Use Partial

Partial types are not a general modularity strategy for everyday domain modeling. If two files exist only because the class is too large, that may indicate the class has too many responsibilities. In other words, partial can organize code, but it should not hide bad design.

See also [extension-methods.md](./extension-methods.md) and [constructors-chaining-and-static.md](./constructors-chaining-and-static.md).

## Code Example

```csharp
using System;

var customer = new Customer { Name = "Mikhail" };
customer.Rename("Misha");
Console.WriteLine(customer.Name); // Misha

public partial class Customer
{
    public string Name { get; set; } = string.Empty;

    public void Rename(string newName)
    {
        OnRenaming(newName); // Hook declared in another partial part.
        Name = newName;
    }

    partial void OnRenaming(string newName);
}

public partial class Customer
{
    partial void OnRenaming(string newName)
    {
        if (string.IsNullOrWhiteSpace(newName))
        {
            throw new ArgumentException("Name cannot be empty.", nameof(newName));
        }
    }
}
```

## Common Mistakes / Pitfalls

- Editing generated partial files instead of adding behavior in a separate hand-written file.
- Assuming partial creates multiple runtime types instead of one merged type.
- Forgetting that non-erasable partial methods must be implemented.
- Using partial as an excuse to keep a class that should really be split into smaller types.
