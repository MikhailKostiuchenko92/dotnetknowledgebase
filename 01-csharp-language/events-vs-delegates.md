# Events vs Delegates

**Category:** C# / Delegates, Events, Lambdas
**Difficulty:** Middle
**Tags:** `event`, `delegate`, `encapsulation`, `EventHandler`, `publisher-subscriber`

## Question

> What is the difference between an `event` and a plain delegate field? Why does the `event` keyword exist, and what does it actually enforce?

Also asked as:
- "Why can't an external class invoke an event defined on another class?"
- "What conventions should you follow when defining and raising events in C#?"

## Short Answer

An `event` is a delegate field with access restrictions enforced by the compiler: only the declaring class can invoke it or replace it outright; outside callers can only subscribe (`+=`) or unsubscribe (`-=`). Without `event`, any caller with access to the field could overwrite all subscribers with `=` or invoke it directly. The `event` keyword enforces the publisher-subscriber pattern and prevents accidental misuse.

## Detailed Explanation

### What `event` Compiles To

A field-like event declaration:

```csharp
public event EventHandler<DataEventArgs> DataReceived;
```

compiles to a private backing delegate field plus two compiler-generated accessor methods:

```csharp
private EventHandler<DataEventArgs> DataReceived;   // private!

public void add_DataReceived(EventHandler<DataEventArgs> value)
    => DataReceived += value;
public void remove_DataReceived(EventHandler<DataEventArgs> value)
    => DataReceived -= value;
```

The `+=` and `-=` operators at call sites route through these accessors. Only `add_` and `remove_` are public; the backing field is private, so:

- External code **cannot** do `source.DataReceived = null` (would reset all subscribers).
- External code **cannot** do `source.DataReceived(this, args)` (would invoke on behalf of the publisher).
- External code **can** only `source.DataReceived += Handler` and `source.DataReceived -= Handler`.

### Plain Delegate Field vs Event

| Capability | Plain `public` delegate field | `public event` |
|---|---|---|
| External `+=` / `-=` | ✅ | ✅ |
| External `=` (overwrite) | ✅ (danger!) | ❌ compile error |
| External direct invocation | ✅ (danger!) | ❌ compile error |
| Internal invocation | ✅ | ✅ |
| Thread-safe add/remove | ❌ manual lock | ✅ lock generated by compiler* |

\* For field-like events the compiler generates a `lock`-based add/remove in .NET Framework; in .NET Core the pattern uses `Interlocked` compare-exchange for better performance.

### EventHandler Convention

The .NET framework convention for events:
1. Return type must be `void`.
2. Two parameters: `object sender` and a class derived from `EventArgs`.
3. Name the event in the present or past tense of what happened: `DataReceived`, `ButtonClicked`, `OrderPlaced`.
4. Name the raising method `OnEventName` (protected virtual in base classes).

```csharp
public class Sensor
{
    public event EventHandler<SensorDataEventArgs>? DataReceived;

    // Protected so derived classes can re-raise
    protected virtual void OnDataReceived(SensorDataEventArgs e)
        => DataReceived?.Invoke(this, e);
}
```

### Custom Event Accessors

You can write explicit `add`/`remove` accessors to control behavior — useful for:
- Locking around subscription for thread safety beyond the default.
- Storing handlers in an `EventHandlerList` (saves memory on objects with many rarely-used events).
- Filtering duplicate subscriptions.

```csharp
private Action<int>? _handler;

public event Action<int> ValueChanged
{
    add    => _handler += value;
    remove => _handler -= value;
}
```

### When to Use a Plain Delegate vs Event

| Use case | Recommendation |
|---|---|
| Callback supplied by the **caller** (strategy, hook) | Plain `Action`/`Func` parameter |
| Notification that **multiple subscribers** may receive | `event` |
| Single callback stored in a field | `Func`/`Action` field (consider `event` for safety) |
| Public API surface (library) | Always `event` for notifications |

> **Rule of thumb:** If the owning class is the only one that should invoke it, use `event`. If the consumer passes a callback for the class to call exactly once, use a delegate parameter.

## Code Example

```csharp
using System;

public class EventArgs<T>(T value) : EventArgs
{
    public T Value { get; } = value;
}

public class TemperatureSensor
{
    // event — subscribers can only += and -=
    public event EventHandler<EventArgs<double>>? TemperatureChanged;

    private double _temperature;

    public double Temperature
    {
        get => _temperature;
        set
        {
            if (Math.Abs(value - _temperature) > 0.1)
            {
                _temperature = value;
                OnTemperatureChanged(new EventArgs<double>(value));
            }
        }
    }

    protected virtual void OnTemperatureChanged(EventArgs<double> e)
        => TemperatureChanged?.Invoke(this, e);   // only the owner can invoke
}

public class Dashboard
{
    public Dashboard(TemperatureSensor sensor)
    {
        sensor.TemperatureChanged += OnTemperatureChanged;
    }

    private void OnTemperatureChanged(object? sender, EventArgs<double> e)
        => Console.WriteLine($"Temperature: {e.Value:F1}°C");
}

// ---- Demo ----
var sensor = new TemperatureSensor();
var dash   = new Dashboard(sensor);

sensor.Temperature = 22.5;   // prints: Temperature: 22.5°C
sensor.Temperature = 22.6;   // no event (delta < 0.1)
sensor.Temperature = 25.0;   // prints: Temperature: 25.0°C

// sensor.TemperatureChanged(sensor, null!);  // ❌ CS0070 — cannot invoke outside class
// sensor.TemperatureChanged = null;          // ❌ CS0070 — cannot assign outside class
```

## Common Mistakes / Pitfalls

- **Using a plain `public` delegate field instead of `event`.** This allows any consumer to erase all subscriptions with `= null` or invoke the delegate directly — violating encapsulation.
- **Not null-checking before raising.** `DataReceived(this, args)` throws if no subscribers. Always use `DataReceived?.Invoke(this, args)` or copy to a local first.
- **Forgetting to unsubscribe.** Subscribers that outlive the publisher hold a reference to the subscriber object, preventing GC. [See: event-memory-leaks.md](./event-memory-leaks.md).
- **Raising events before object construction is complete.** If a virtual `OnEventName` is called from a constructor and a derived class overrides it, the derived class's fields may not yet be initialized.
- **Using non-standard event signatures.** Always `(object? sender, TEventArgs e)` for public APIs; non-standard signatures confuse tooling (e.g., WinForms designer, event wiring generators).
