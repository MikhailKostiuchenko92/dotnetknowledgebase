# Tiered Compilation

**Category:** .NET Runtime / JIT & AOT
**Difficulty:** 🟡 Middle-Senior
**Tags:** `tiered compilation`, `Tier 0`, `Tier 1`, `QuickJit`, `OSR`, `call counting`, `DOTNET_TC_QuickJitForLoops`

## Question

> What is tiered compilation in .NET, and why does the runtime compile the same method more than once?

Also asked as:
> What are Tier 0, Tier 1, QuickJit, and OSR?
> How does the runtime decide when a method is hot enough to recompile?

## Short Answer

Tiered compilation is the runtime strategy of generating a quick, minimally optimized version of a method first and then recompiling hot methods with better optimizations later. Tier 0 improves startup and first-use latency, while Tier 1 improves steady-state throughput. The runtime uses call counting and hotness heuristics to trigger promotion, and newer runtimes can even perform on-stack replacement (OSR) so long-running loops switch to optimized code mid-execution.

## Detailed Explanation

### Why Tiering Exists

Classic JIT compilation faces a trade-off: spend more time optimizing now, or start executing sooner. Tiered compilation avoids choosing just one side. It emits cheap code first, then invests optimization work only in methods that prove to be hot.

That makes tiering a major reason modern .NET starts faster than older “optimize everything immediately” approaches while still delivering strong throughput for long-running services.

### Tier 0 vs Tier 1

| Tier | Goal | Characteristics |
|---|---|---|
| Tier 0 | Fast startup | Quick code generation, limited optimization |
| Tier 1 | Better throughput | More expensive optimizations for hot code |

Tier 0 is often generated by **QuickJit**, which prioritizes low compilation cost. Once the runtime sees enough usage, it schedules a better-optimized Tier 1 version.

A common rule of thumb is that a method becomes eligible for promotion after roughly **30 calls** in current runtimes, but that number is a heuristic, not a contract. Interviewers usually want you to understand the idea of call counting rather than recite a fixed magic number.

### Call Counting and Promotion

The runtime instruments or tracks method entry counts to detect hot methods. When the threshold is hit, it queues recompilation in the background. The next calls can then use the optimized Tier 1 body.

This matters because most methods in a real application are not hot. Tiering keeps the runtime from wasting expensive optimization effort on cold code paths.

### QuickJit and QuickJitForLoops

**QuickJit** makes startup better by emitting simple machine code quickly. There is also a configuration switch, `DOTNET_TC_QuickJitForLoops`, that controls whether methods containing loops are eligible for quick Tier 0 compilation.

Why is that interesting? Loops are often the hottest part of an application. In some scenarios, you may prefer a method with loops to wait for better optimization; in others, you want it to start immediately and then rely on OSR to improve it later.

### On-Stack Replacement (OSR)

OSR, introduced in modern form in .NET 7+, allows a method that is **already running inside a long loop** to switch from Tier 0 code to optimized Tier 1 code without waiting for the method to return and be called again.

That is especially valuable for workloads like:

- long-running data processing loops
- server-side request handlers with heavy iteration
- benchmarks and numerical code

Without OSR, a huge loop might spend its whole lifetime in mediocre Tier 0 code. With OSR, the runtime can patch execution into a better loop body after hotness is detected.

> **Warning:** Tiering, QuickJit, and OSR are runtime heuristics. They are great for real applications, but microbenchmarks can become misleading if you do not warm up the code first.

### Practical Tuning

You can influence behavior with runtime configuration or environment variables such as tiered-compilation settings and `DOTNET_TC_QuickJitForLoops`, but this is advanced tuning. The default heuristics are usually good enough.

For most interview answers, say: the runtime optimizes hot methods progressively, which improves both startup and throughput compared with a one-shot JIT model.

### Interview Takeaway

A complete answer is: Tier 0 gets code running fast, Tier 1 optimizes hot code, call counting drives promotion, QuickJit keeps startup responsive, and OSR lets long-running loops switch to better machine code while they are still executing. See [jit-compilation-basics.md](./jit-compilation-basics.md) and [on-stack-replacement.md](./on-stack-replacement.md).

## Code Example

```csharp
using System.Diagnostics;
using System.Runtime.CompilerServices;

namespace RuntimeSamples;

public static class TieredCompilationDemo
{
    public static void Main()
    {
        // Warm the method enough times that the runtime may promote it to Tier 1.
        for (int i = 0; i < 100; i++)
        {
            ComputeHotPath(i);
        }

        Stopwatch sw = Stopwatch.StartNew();
        long total = 0;

        for (int i = 0; i < 5_000_000; i++)
        {
            total += ComputeHotPath(i);
        }

        sw.Stop();
        Console.WriteLine($"Total={total}, elapsed={sw.ElapsedMilliseconds} ms");
        Console.WriteLine("Try toggling DOTNET_TieredCompilation or DOTNET_TC_QuickJitForLoops to observe behavior.");
    }

    [MethodImpl(MethodImplOptions.NoInlining)]
    private static int ComputeHotPath(int value) => (value * 17) ^ (value >> 2);
}
```

## Common Mistakes / Pitfalls

- Describing tiered compilation as if a method is JIT-compiled only once.
- Treating the promotion threshold as a fixed contractual number.
- Ignoring OSR when discussing long-running loops on modern runtimes.
- Benchmarking cold Tier 0 code and assuming the result represents steady-state performance.
- Disabling tiering globally without a measured reason.
