# Design a Payment System

**Category:** System Design / Classic Problems
**Difficulty:** Senior
**Tags:** `payments`, `idempotency`, `ledger`, `exactly-once`, `reconciliation`, `pci-dss`

## Question

> Design a payment processing system for an e-commerce platform. The system must charge customers, pay out to sellers, handle refunds, and process 1000 transactions per second. Correctness is non-negotiable — no double charges, no lost money.

- How do you guarantee exactly-once payment execution?
- How do you model money (the ledger)?
- What happens when a third-party payment gateway times out?

## Short Answer

Every payment operation is assigned a globally unique idempotency key by the client. The payment service persists the request and its result together in a single database transaction before calling the external gateway; if the gateway call fails or times out, a retry with the same idempotency key returns the cached result rather than re-charging. Money is never stored as a floating-point balance — it uses an immutable double-entry ledger of integer amounts (pence/cents). Reconciliation jobs compare our ledger against the gateway's settlement reports nightly to detect discrepancies.

## Detailed Explanation

### Correctness Requirements (Non-Negotiable)

| Guarantee | Mechanism |
|-----------|-----------|
| No double charge | Idempotency key checked before every gateway call |
| No lost money | Double-entry ledger — debits always equal credits |
| Atomicity | Payment request + ledger entry in one DB transaction |
| Auditability | Immutable append-only ledger; never UPDATE rows |
| Reconciliation | Nightly comparison with gateway settlement CSV |

### Idempotency — The Foundation

An idempotency key is a UUID generated by the **client** and included with every payment request. The server:

1. Checks if this key already exists in `payment_requests`.
2. If yes → return the stored response (success or failure). Do NOT re-charge.
3. If no → persist the request (status=`PENDING`), call the gateway, update status, return response.

The key is stored with a result hash. On retry, if the request parameters differ from the original (key re-use attack), return `HTTP 422 Unprocessable Entity`.

```
Client retries (timeout)
        │ idempotency_key = "pay_abc123"
        ▼
Payment Service
  ├── EXISTS pay_abc123 in DB? → YES: return stored result (charge already happened or definitively failed)
  └── NO: persist request, call gateway, store result
```

### Double-Entry Ledger

Never update balances. Instead, record every money movement as a pair of ledger entries (one debit, one credit). The account balance is always computed as `SUM(credits) - SUM(debits)`.

```sql
CREATE TABLE ledger_entries (
    id              BIGSERIAL PRIMARY KEY,
    transaction_id  UUID NOT NULL,
    account_id      UUID NOT NULL,
    entry_type      VARCHAR(8) NOT NULL,  -- 'DEBIT' | 'CREDIT'
    amount_cents    BIGINT NOT NULL CHECK (amount_cents > 0),
    currency        CHAR(3) NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    description     TEXT
);

-- Payment: debit customer account, credit merchant escrow
INSERT INTO ledger_entries VALUES
    ('txn-001', 'customer-account-id', 'DEBIT',  4999, 'USD', NOW(), 'Order #789'),
    ('txn-001', 'escrow-account-id',   'CREDIT', 4999, 'USD', NOW(), 'Order #789');
-- These two rows ALWAYS INSERT together in one transaction → perfect balance
```

> **Warning:** Never store money as `FLOAT` or `DOUBLE`. Floating-point rounding errors will eventually produce $0.00000001 discrepancies that are invisible until reconciliation. Always use `BIGINT` cents.

### Payment Flow (State Machine)

```
INITIATED → PENDING → PROCESSING → COMPLETED
                                 ↘ FAILED
                                 ↘ REFUNDED
```

State transitions are the only allowed mutations; the state history is appended to `payment_events` (event sourcing pattern).

### Handling Gateway Timeouts

A gateway timeout is the hardest case: we don't know if the charge happened.

Strategy — **Check-then-confirm**:
1. Gateway call times out after 5 s.
2. Payment service marks status = `UNKNOWN`.
3. A background reconciliation job polls the gateway's "transaction status" endpoint with our reference ID (retry every 60 s, up to 24 h).
4. When gateway confirms: update status = `COMPLETED` or `FAILED`; post ledger entries accordingly.
5. During `UNKNOWN` window: block re-submission with the same order ID.

### PSP (Payment Service Provider) Abstraction

Wrap Stripe / Braintree / Adyen behind a common interface. This allows switching providers or routing by currency/region:

```csharp
public interface IPaymentGateway
{
    Task<GatewayResult> ChargeAsync(ChargeRequest request, CancellationToken ct);
    Task<GatewayResult> RefundAsync(RefundRequest request, CancellationToken ct);
    Task<TransactionStatus> GetStatusAsync(string gatewayReference, CancellationToken ct);
}
```

### Reconciliation

Nightly job:
1. Download settlement CSV from PSP (each charge has PSP reference + amount).
2. Join against our `payment_requests` table on PSP reference.
3. Discrepancies → alert on-call engineer:
   - In our DB as `COMPLETED`, not in PSP file → **lost charge** (critical)
   - In PSP file, not in our DB → **ghost charge** (critical)
   - Amount mismatch → **fee dispute**

### PCI DSS Compliance

- Never log or store raw PANs (card numbers), CVVs, or expiry in your systems.
- Use PSP tokenisation: client sends card data directly to PSP (hosted fields / Stripe Elements); PSP returns a one-time token; your server charges using the token.
- Your servers never touch raw card data → reduces PCI scope to SAQ A (simplest).
- Encrypt all payment-related data at rest (AES-256); enforce TLS 1.2+ for all API traffic.

### Capacity

| Metric | Value |
|--------|-------|
| Peak TPS | 1000 |
| Avg payment size | 1 KB |
| Ledger writes/s | 2000 (debit + credit per payment) |
| Database | PostgreSQL (ACID required); read replicas for reporting |

## Code Example

```csharp
using Microsoft.EntityFrameworkCore;

namespace Payments;

public sealed class PaymentService(PaymentsDbContext db, IPaymentGateway gateway)
{
    public async Task<PaymentResult> ProcessAsync(
        PaymentRequest request, CancellationToken ct = default)
    {
        // 1. Check idempotency — look up existing payment by idempotency key
        var existing = await db.PaymentRequests
            .FirstOrDefaultAsync(p => p.IdempotencyKey == request.IdempotencyKey, ct);

        if (existing is not null)
            return existing.ToResult(); // replay cached result — DO NOT re-charge

        // 2. Persist request in PENDING state (inside transaction)
        await using var txn = await db.Database.BeginTransactionAsync(ct);

        var payment = new PaymentRequest
        {
            Id             = Guid.NewGuid(),
            IdempotencyKey = request.IdempotencyKey,
            OrderId        = request.OrderId,
            AmountCents    = request.AmountCents,
            Currency       = request.Currency,
            Status         = PaymentStatus.Pending,
            CreatedAt      = DateTimeOffset.UtcNow,
        };
        db.PaymentRequests.Add(payment);
        await db.SaveChangesAsync(ct); // fails if duplicate idempotency key (unique index)
        await txn.CommitAsync(ct);

        // 3. Call external gateway (outside DB transaction — keeps connection time short)
        GatewayResult gatewayResult;
        try
        {
            gatewayResult = await gateway.ChargeAsync(
                new ChargeRequest(payment.Id, request.AmountCents, request.Currency, request.Token),
                ct);
        }
        catch (TaskCanceledException)
        {
            // Timeout → unknown state; reconciliation will resolve
            await UpdateStatusAsync(payment.Id, PaymentStatus.Unknown, null, ct);
            return new PaymentResult(payment.Id, PaymentStatus.Unknown);
        }

        // 4. Record gateway result + post double-entry ledger entries atomically
        await using var txn2 = await db.Database.BeginTransactionAsync(ct);
        await UpdateStatusAsync(payment.Id, gatewayResult.Success
            ? PaymentStatus.Completed : PaymentStatus.Failed,
            gatewayResult.Reference, ct);

        if (gatewayResult.Success)
        {
            // Double-entry: debit customer, credit merchant escrow
            db.LedgerEntries.AddRange(
                new LedgerEntry(Guid.NewGuid(), payment.Id, request.CustomerAccountId,
                    EntryType.Debit,  payment.AmountCents, payment.Currency),
                new LedgerEntry(Guid.NewGuid(), payment.Id, request.MerchantEscrowId,
                    EntryType.Credit, payment.AmountCents, payment.Currency));
        }

        await db.SaveChangesAsync(ct);
        await txn2.CommitAsync(ct);

        return new PaymentResult(payment.Id,
            gatewayResult.Success ? PaymentStatus.Completed : PaymentStatus.Failed);
    }

    private async Task UpdateStatusAsync(
        Guid id, PaymentStatus status, string? reference, CancellationToken ct)
    {
        await db.PaymentRequests
            .Where(p => p.Id == id)
            .ExecuteUpdateAsync(s => s
                .SetProperty(p => p.Status, status)
                .SetProperty(p => p.GatewayReference, reference), ct);
    }
}
```

## Common Mistakes / Pitfalls

- **Storing money as `decimal` or `float`**: use `BIGINT` cents exclusively; all display formatting happens in the presentation layer.
- **Calling the payment gateway inside a DB transaction**: a long-running gateway call (300–800 ms) inside a transaction holds DB locks and exhausts the connection pool.
- **Generating idempotency keys server-side**: the server cannot distinguish a legitimate retry from a new request if it generates the key; the client must generate and send it.
- **Not modelling the `UNKNOWN` state**: ignoring gateway timeouts leads to double charges on client retry.
- **Mutable balance rows instead of a ledger**: a single UPDATE on a balance row under high concurrency produces race conditions and requires heavy locking.
- **Logging payment tokens or gateway responses verbatim**: gateway responses often contain sensitive PAN fragments; scrub before logging.
